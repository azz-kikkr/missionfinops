Enable AWS Data Exports in 5 minutes.

Your billing history is not being preserved. Fix that today. No Athena, no Glue, no architecture diagram. Just data flowing to S3.

Why do this?

  • AWS Cost Explorer gives you summaries. Data Exports gives you the raw billing data.
  • When someone asks "what changed last month?", you will already have the history.
  • Every FinOps tool, pipeline, and query engine starts here. This is the foundation.

Let's enable it.

Before you start

You need two things:

  • Access to the Billing & Cost Management console (requires billing permissions on the management account, or a delegated admin)
  • An S3 bucket to receive the export (or let AWS create one for you)

That's it. No Athena. No Glue. No Lake Formation. Those come later if you want them.

Step 1: Open Data Exports

In the AWS Console, navigate to:

Billing & Cost Management → Data Exports → Create

You will see the Data Exports landing page with a blue "Create" button. Click it.

Note: If you see "Cost and Usage Reports" in the sidebar instead, that is the legacy page. AWS is deprecating it. Look for "Data Exports" under Cost and Usage Analysis. The new experience gives you CUR 2.0 with a better schema and Parquet support out of the box.

Step 2: Choose Standard Data Export + CUR 2.0

On the "Create export" page, you will see three export types at the top:

  • Standard data export (select this one)
  • Cost and usage dashboard powered by QuickSight (skip)
  • Legacy CUR export (skip)

Under "Data table content settings", select CUR 2.0.

Why CUR 2.0, not FOCUS? CUR 2.0 is the native AWS terminology. It is what most people recognize, what most FinOps tools expect, and what is easiest to Google when you get stuck. FOCUS is excellent and we will cover it in the next post in this series.

Give your export a name. Something like MyCostExport or billing-daily. Keep it simple.

Step 3: Recommended settings

Under "Additional export content", check these two boxes:

  • Include resource IDs — so you can trace costs back to individual resources later
  • Split cost allocation data — so shared costs (ECS, EKS) are broken out by container

Under "Report data processing settings":

  • Refresh automatically — check this. AWS will update the report when refunds, credits, or support fees are applied retroactively.
Tip: If you enable "Split cost allocation data", you also need to opt in to that feature under Cost Management Preferences. AWS will show you a blue info banner reminding you. Do it now or the column will be empty.

Step 4: Delivery settings

Keep it simple:

  • Time granularity: Daily (gives you daily-level cost rows)
  • File versioning: Overwrite existing data export file (saves S3 storage costs)
  • Report data integration: leave all unchecked (no Athena, no Redshift, no QuickSight)
  • Compression and format: Parquet

Why these choices? Because this guide is about collecting data. Querying it is the next step. Parquet is columnar, compressed, and what every modern tool expects. You will thank yourself later.

Step 5: S3 destination

Choose your bucket and prefix:

  • Storage account: This account
  • S3 bucket: your bucket name (e.g. mycompany-billing)
  • S3 path prefix: something like cur/

If you do not have a bucket yet, click "Configure" and AWS will walk you through creating one with the correct bucket policy. The policy allows only the AWS billing service to write to it.

Click Create.

Step 6: Verify

Wait 24 hours. AWS delivers the first export within 24 hours of creation.

Then check your bucket:

s3://mycompany-billing/cur/

You should see Parquet files appearing. That's it. Your billing history is now being preserved.

What's next?

You have done the hardest part. Your billing data is now being collected and preserved in S3. Most teams never get this far because it seems like it should be harder than it is.
